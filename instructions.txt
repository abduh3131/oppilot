# OpenPilot Mock Sensor Stack – Quick Start

These instructions assume you want to work from Windows with WSL, with this
repository kept inside an `OPP` folder and the OpenPilot checkout living
directly under your Windows user directory:

```
C:\Users\Abduh\
├── OPP\
│   └── oppilot\        (this repository)
└── openpilot\          (commaai/openpilot checkout)
```

> 📝 Tip: clone `commaai/openpilot` into `C:\Users\Abduh\openpilot\` and this
> repository into `C:\Users\Abduh\OPP\oppilot\` before you begin. The
> commands below use those paths.

---

## 1. Prepare WSL

1. Open **Windows Terminal** → **Ubuntu** (WSL).
2. Verify the project folders are mounted:
   ```bash
   ls /mnt/c/Users/Abduh/OPP
   ls /mnt/c/Users/Abduh/openpilot
   ```
   You should see `oppilot` under the first path and the OpenPilot files under
   the second.

---

## 2. Set up the mock sensor environment

All commands in this section are executed **inside WSL**.

1. Move into the adapter project and create/activate the virtual environment:
   ```bash
   cd /mnt/c/Users/Abduh/OPP/oppilot/op-jetson-adapters
   python3 -m venv .venv
   source .venv/bin/activate
   pip install -U pip
   pip install -r requirements.txt numpy pillow
   ```

2. Start the mock sensor stack (cameras + GPS):
   ```bash
   ./tools/run_all.sh
   ```
   Leave this terminal open; it supervises the background publishers. You
   should see log lines if any sensor subprocess exits unexpectedly.

3. (Optional) In a **second** WSL terminal, you can inspect the ZeroMQ bus:
   ```bash
   cd /mnt/c/Users/Abduh/OPP/oppilot/op-jetson-adapters
   source .venv/bin/activate
   python tools/sniffer.py
   ```
   You will see `gps.raw`, `camera.front`, and `camera.front.bytes` messages as
   they stream.

---

## 3. Prepare the OpenPilot replay tools

These commands are also run in **WSL** but from the `openpilot` checkout.

1. Install OpenPilot dependencies (first run only):
   ```bash
   cd /mnt/c/Users/Abduh/openpilot
   python3 -m pip install -U pip
   pip install -r requirements_openpilot.txt  # generates the replay venv deps
   ```
   > ℹ️ The exact requirements file name can differ across OpenPilot releases
   > (e.g., `requirements.txt` plus `requirements_optional.txt`). Follow the
   > upstream README if the file is renamed.

2. Build OpenPilot (first run only):
   ```bash
   scons -j$(nproc)
   ```

3. Download or copy a comma recording (route) into `~/openpilot_data/`.
   For example, place the route `a2b3c4d5e6f7abcd|2023-05-01--12-34-56` inside
   that folder.

---

## 4. Run an OpenPilot replay against the mock sensors

1. In a new WSL terminal:
   ```bash
   cd /mnt/c/Users/Abduh/openpilot
   source tools/replay/plotjuggler_venv/bin/activate  # or the venv created above
   ./tools/replay/replay.py --demo --qcamera --dcam --no-controller --route <ROUTE_NAME>
   ```
   Replace `<ROUTE_NAME>` with the route you downloaded. The `--demo`
   flag tells replay to consume the log without sending commands to a car.

2. Ensure the mock sensor stack from step 2 is still running. Replay uses
   OpenPilot’s logged data for vehicle state while the adapters provide the
   simulated live camera and GPS streams.

3. To stop everything:
   - Press `Ctrl+C` in the replay terminal.
   - Press `Ctrl+C` in the `tools/run_all.sh` terminal. The supervisor will
     send SIGINT to each mock publisher for a clean shutdown.

---

## 5. Troubleshooting

- **No frames/GPS data?** Run `python tools/sniffer.py` to confirm the mock
  publishers are active. If nothing appears, restart `./tools/run_all.sh`.
- **Port already in use?** Another ZeroMQ publisher might still be bound to
  `tcp://127.0.0.1:5557`. Stop stale processes with
  `pkill -f adapters.mock_camera_publisher` and `pkill -f adapters.gpsd_bridge`.
- **Python package errors?** Re-run the virtual environment setup commands.
- **Different Windows username or path?** Adjust the `/mnt/c/...` paths to match
  your system.

These steps give you a self-contained simulation loop: mock sensors feed live
ZeroMQ topics while OpenPilot replays previously recorded drives for testing and
UI exploration.
